# Benchmark snapshot — 2026-05-16

First published numbers from the re-architected suite: four parallel-consumer runtimes against a
Testcontainers-managed Kafka broker (4.2.0). Three of four runtimes produced clean JMH scores
across the full `workMicros` sweep; one (Reactor Kafka 1.3.23) crashed at runtime against
`kafka-clients:4.2.0` and has no usable data yet.

## Throughput — `ParallelProcessingBenchmark`

Records / second, higher is better. Each cell is `Score ± error` from `Cnt=10` (5 iterations × 2
forks). Reactor Kafka cells crashed (see "Pitfalls"); raw `KafkaConsumer + VT` is the framework-free
baseline.

| Runtime | `workMicros=0` | `workMicros=100` | `workMicros=1000` |
|---|---:|---:|---:|
| **KPipe** | **553,704 ± 54,924** | **500,903 ± 95,758** | **495,803 ± 41,069** |
| **Raw `KafkaConsumer` + VT** | 570,890 ± 38,369 | 526,568 ± 44,444 | 498,016 ± 51,323 |
| **Confluent Parallel Consumer** | 107,877 ± 8,892 | 112,985 ± 2,653 | 66,454 ± 2,963 |
| **Reactor Kafka** | — (crashed) | — (crashed) | — (crashed) |

### Observations

- **KPipe ≈ raw `KafkaConsumer + VT` across the whole sweep**, within the error bars. KPipe's
  framework overhead on the consumer hot path is statistically invisible in this run. The
  framework wraps the loop without slowing it.
- **Both Loom runtimes are 4.4× faster than Confluent Parallel Consumer at `workMicros=0`** and
  the gap widens to **7.5× at `workMicros=1000`** where Confluent's 100-worker thread pool starts
  blocking on the `parkNanos` simulating I/O. KPipe's virtual-thread-per-record model parks each
  record's carrier-thread continuation without consuming a platform thread; Confluent's pool
  serialises through 100 platform-thread slots.
- **Confluent has almost no `workMicros=0 → workMicros=100` falloff** (108k → 113k, within error)
  because the pool isn't saturated yet. The drop at `workMicros=1000` (113k → 66k) is the
  platform-thread pool saturating. The KPipe and raw curves are flatter because virtual threads
  scale beyond the partition count.

## Allocation profile (`gc.alloc.rate.norm`, B / op)

Lower is better. JMH's GC profiler reports allocation per benchmark operation (= per record).

| Runtime | `workMicros=0` | `workMicros=100` | `workMicros=1000` |
|---|---:|---:|---:|
| KPipe | 923 | 1,530 | 1,484 |
| Raw `KafkaConsumer` + VT | 55 | 455 | 431 |
| Confluent Parallel Consumer | 34 | 34 | 34 |

### Observations

- **Confluent allocates least** (~34 B/op) and is consistent across the workload sweep — its
  pool of long-lived worker threads doesn't allocate much per record.
- **Raw is the second-leanest** (55–455 B/op). The `workMicros=0 → 100` jump is allocation in the
  virtual-thread Runnable lambdas piling up under blocking work.
- **KPipe allocates most** (923–1530 B/op) because of `Result<T>` wrappers, the pipeline's
  per-record builder hand-off, and a fresh virtual thread per record. **At the throughput levels
  shown above, that allocation rate is not a bottleneck** — the JVM's young gen handles it
  without GC pause causing a throughput dip. If you're memory-constrained or running on a smaller
  heap, this is the number to watch.

## GC behavior

`gc.count` (events) and `gc.time` (total ms across measurement iterations).

| Runtime | `workMicros=0` count / ms | `workMicros=100` count / ms | `workMicros=1000` count / ms |
|---|---|---|---|
| KPipe | 99 / 132 | 137 / 247 | 132 / 420 |
| Raw `KafkaConsumer` + VT | 94 / 110 | 109 / 165 | 107 / 261 |
| Confluent Parallel Consumer | 347 / 628 | 354 / 473 | 314 / 602 |

Counter-intuitive but consistent with the throughput numbers: **Confluent has the fewest bytes
per op but the most GC events overall**, because its iterations take longer (more wall-clock
time, more young-gen cycles). KPipe and raw flush their gen faster but each iteration ends
sooner so the cumulative GC time is lower.

## Pitfalls in this run

- **Reactor Kafka 1.3.23 crashed at runtime.** Stack trace excerpt:

  ```
  java.lang.NoSuchMethodError: 'void org.apache.kafka.clients.consumer.ConsumerRecord.<init>(
    java.lang.String, int, long, long, org.apache.kafka.common.record.TimestampType,
    java.lang.Long, int, int, java.lang.Object, java.lang.Object,
    org.apache.kafka.common.header.Headers)'
        at reactor.kafka.receiver.ReceiverRecord.<init>(ReceiverRecord.java:48)
  ```

  Reactor Kafka 1.3.x targets `kafka-clients:3.7.x` and the `ConsumerRecord` constructor shape
  changed in 4.x. Need to either upgrade to a Reactor Kafka build that supports kafka-clients
  4.x (none stable at time of writing), or pin the bench to a 3.x client for the Reactor row
  only. Tracked as a follow-up.

- **Single fork would have masked variance.** The `±54,924` error band on KPipe@0 is real — JMH
  ran two forks × five iterations, and there's iteration-to-iteration spread driven by
  container scheduler noise. Tighter numbers would require pinning the broker container to a
  set of cores.

- **Docker overhead is real.** First iteration of every trial pays container startup. JMH's 3
  warmup iterations absorbed it cleanly here; smaller warmup counts would skew the first
  measurement.

- **No latency-percentile run yet.** The `ParallelProcessingLatencyBenchmark` was not exercised
  in this run. Coming in a follow-up snapshot.
