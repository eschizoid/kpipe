## Throughput — ParallelProcessingBenchmark

Records / second, higher is better. `workMicros` is per-record simulated work via `LockSupport.parkNanos`. Ordering
column matters: only KEY_ORDERED guarantees per-key order; every other arm is unordered.

| Runtime                     | Ordering    |      workMicros=0 |   workMicros=100 |  workMicros=1000 |
| --------------------------- | ----------- | ----------------: | ---------------: | ---------------: |
| Raw `KafkaConsumer` + VT    | none        |  630,515 ± 39,934 | 573,101 ± 42,405 | 573,443 ± 41,924 |
| KPipe (PARALLEL)            | none        | 525,051 ± 159,972 | 581,500 ± 45,676 | 503,612 ± 86,316 |
| KPipe (KEY_ORDERED)         | **per-key** |  311,854 ± 62,037 | 240,005 ± 76,572 |   89,745 ± 1,056 |
| Reactor Kafka               | none        |  309,724 ± 15,480 |     78,953 ± 652 |      9,011 ± 106 |
| Confluent Parallel Consumer | none        |   111,541 ± 3,907 | 108,518 ± 12,318 |   64,539 ± 8,321 |
| Kafka Share Consumer (932)  | none        |        4,839 ± 22 |       4,813 ± 21 |       4,789 ± 38 |

### Observations

- **Raw VT is the ceiling; KPipe PARALLEL tracks it within ~10%.** Both lean on Loom and stay flat-to-rising across
  `workMicros` — virtual threads absorb blocking work without a thread-pool ceiling. KPipe's gap below raw is the cost
  of offset management + the pipeline, and it's small.
- **KEY_ORDERED shows the ordering tax cleanly.** 312k → 90k as work rises: per-key serial queues bound intra-key
  parallelism, so 1 ms of blocking work per record can't be hidden the way PARALLEL hides it. It still beats Confluent
  and Reactor at `workMicros=1000`. Read this as "what ordering costs," not as a loss to the unordered arms.
- **Reactor collapses under blocking work** (310k → 9k). `Flux.parallel(N).runOn(Schedulers.parallel())` is
  CPU-core-bound; 1 ms blocking per record on a core-bound scheduler is the classic reactive-blocking pitfall
  (`boundedElastic` would change this, but that's not the idiomatic parallel path).
- **Confluent is pool-bound** (~111k → ~64k). 100 platform workers cap throughput once each record blocks for 1 ms.
- **Share is protocol-bound and flat** at ~4.8k regardless of `workMicros`. The per-record work — even 1 ms — is dwarfed
  by the share-acquire + acknowledge + `__share_group_state` round-trips. **This is one share member.** Share's whole
  point is horizontal fan-out across many members beyond partition count; a single-process bench does not exercise that,
  so read 4.8k as "one member's ceiling in early KIP-932," **not** "share consumers are slow." A multi-member experiment
  is the honest way to evaluate share throughput.

## Allocation profile (GC profiler)

`gc.alloc.rate.norm` (B / op, lower is better) at `workMicros=100`.

| Runtime                     |    B/op |
| --------------------------- | ------: |
| Confluent Parallel Consumer |    34.4 |
| Reactor Kafka               |    84.1 |
| Raw `KafkaConsumer` + VT    |   454.8 |
| Kafka Share Consumer (932)  |   457.2 |
| KPipe (KEY_ORDERED)         | 1,634.1 |
| KPipe (PARALLEL)            | 1,702.1 |

KPipe allocates the most per record (~1.7 KB/op) — the per-record virtual thread + `Result` wrapping + offset tracking.
That's the throughput-vs-allocation trade-off: it buys the highest sustained throughput of the framework arms but at the
highest allocation rate. Confluent's pooled workers allocate almost nothing per record. (Share's low figure is an
artifact of how little it processes per second, not efficiency.)

## Pitfalls in this run

- **Intel x86, not Apple Silicon.** This ran on an i7-8850H (2018, 6c/12t). Loom scheduling and absolute throughput
  differ from Apple Silicon / Linux server hardware — do **not** compare these absolute numbers against runs on other
  machines. The within-run ordering between runtimes is the meaningful signal.
- **`workMicros=0` is the noisiest cell.** KPipe PARALLEL shows ±160k at 0 µs — pure-overhead measurements amplify JIT
  and Loom-scheduling variance. The 100 µs / 1000 µs cells are tighter and more representative.
- **Single share member.** As above — this run says nothing about share-group scale-out, which is share's headline
  feature. Treat the share row as a single-member ceiling, not a verdict.
- **macOS + Docker loopback broker.** Absolute records/sec are lower than a dedicated network broker on server hardware;
  read orderings, not absolutes.
